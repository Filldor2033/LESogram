### Техническая документация (Technical Documentation)

**Название проекта:** LESogram
**Версия:** 1.0.0
**Описание:** Веб-чат в реальном времени с поддержкой комнат, аутентификацией и обменом файлами.

#### 1. Архитектура системы

Проект представляет собой монолитное веб-приложение.

*   **Клиентская часть (Frontend):** Single Page Application (SPA) на чистом JavaScript, HTML5 и CSS3. Располагается в директории `/static`.
*   **Серверная часть (Backend):** Приложение на `FastAPI`. Обрабатывает HTTP-запросы и WebSocket-соединения.
*   **База данных:** `SQLite` для хранения данных о пользователях, комнатах и сообщениях. Взаимодействие через `SQLAlchemy` ORM.
*   **Аутентификация:** Основана на JWT-токенах. Для доступа к эндпоинтам API требуется `Bearer`-токен. Для доступа к room-специфичным эндпоинтам и WebSocket используется отдельный `room_token`.
*   **Реальный время:** Реализовано через WebSocket. Сервер отслеживает подключенных клиентов для каждой комнаты и транслирует новые сообщения и системные события.

---

#### 2. Модель базы данных

База данных SQLite содержит три основные таблицы:

*   **`users`:** Пользователи.
    *   `id` (Integer, PK): Уникальный идентификатор.
    *   `username` (String[50], Unique, Index): Логин пользователя.
    *   `hashed_password` (String[255]): Хеш пароля.
    *   `is_admin` (Boolean, Default: False): Флаг администратора.
*   **`rooms`:** Комнаты.
    *   `id` (Integer, PK): Уникальный идентификатор.
    *   `name` (String[100], Unique, Index): Название комнаты.
    *   `password_hash` (String[255]): Хеш пароля для входа в комнату.
    *   `created_by` (String[50]): Имя создателя комнаты.
    *   `created_at` (DateTime): Дата и время создания.
*   **`messages`:** Сообщения.
    *   `id` (Integer, PK): Уникальный идентификатор.
    *   `username` (String[50], Index): Имя отправителя.
    *   `room` (String[100], Index): Комната, в которую отправлено сообщение.
    *   `text` (String[1000]): Текст сообщения.
    *   `content_type` (String[20], Default: "text"): Тип контента (`text`, `image`, `video`, `file`).
    *   `media_url` (String[500], Nullable): Путь к загруженному файлу.
    *   `file_name` (String[255], Nullable): Оригинальное имя файла.
    *   `mime_type` (String[255], Nullable): MIME-тип файла.
    *   `file_size` (Integer, Nullable): Размер файла в байтах.
    *   `timestamp` (DateTime): Время отправки сообщения.

**Управление схемой БД:**
При запуске приложения `SQLAlchemy` создает таблицы, если их нет. Функции `ensure_user_schema()` и `ensure_message_schema()` в `main.py` реализуют механизм миграций "на лету", добавляя новые колонки в существующие таблицы.

#### 3. Аутентификация и авторизация

**Процесс:**
1.  **`POST /register`** или **`POST /login`**: При успехе сервер возвращает `access_token` (JWT), подписанный с помощью `SECRET_KEY`.
2.  **Доступ к API**: Клиент передает `access_token` в заголовке `Authorization: Bearer <token>`. Функция `get_current_user` в `main.py` декодирует и проверяет его.
3.  **Доступ к комнате**: Для входа в комнату клиент отправляет ее пароль на **`POST /rooms/join`** и получает `room_token`. Этот токен содержит в payload имя пользователя (`sub`), имя комнаты (`room`) и тип (`room_access`). Он используется для авторизации WebSocket-соединений и доступа к истории сообщений/вложениям через HTTP.

**Компоненты:**
*   `auth.py`: Содержит функции `create_access_token` и `verify_token`. Использует библиотеки `python-jose` и `pwdlib`.
*   `main.py`:
    *   `get_current_user`: Извлекает и проверяет `access_token` для HTTP.
    *   `verify_room_token`: Проверяет `room_token` для доступа к ресурсам комнаты.
    *   `get_current_user_model`: Возвращает ORM-модель `User` для эндпоинтов, где нужны полные данные пользователя (например, `is_admin`).

#### 4. API: HTTP и WebSocket

**HTTP-эндпоинты** принимают и возвращают JSON. Структура запросов/ответов валидируется Pydantic-моделями из `schemas.py`.

**WebSocket-эндпоинт:**
*   **URL:** `/ws/{room}?room_token=<token>`
*   **Поведение:**
    *   Проверяет `Origin`.
    *   Проверяет `room_token`. При неудаче закрывает соединение с кодом `1008`.
    *   При подключении пользователь добавляется в `ConnectionManager` для комнаты.
    *   **Отправка:** Клиент отправляет JSON: `{"text": "сообщение"}`.
    *   **Получение:** Клиент получает JSON трех типов:
        1.  `type: "message"` — обычное сообщение.
        2.  `type: "system"` — системное событие (`joined`, `left`, `room_deleted`, `rate_limited`).
        3.  `type: "message_deleted"` — уведомление об удалении сообщения администратором.
    *   **Отключение:** При выходе пользователя (`WebSocketDisconnect`) его счетчик в `room_members` уменьшается. Если он достиг нуля, всем участникам рассылается событие `left`.

**Rate Limiting:**
Реализован через класс `SlidingWindowRateLimiter` в `main.py`. Использует алгоритм скользящего окна. Лимиты применяются как к HTTP-запросам, так и к WebSocket-сообщениям. Ключ для уникальной идентификации клиента строится на основе IP-адреса и, для WebSocket, имени пользователя и комнаты.

#### 5. Безопасность при загрузке файлов

Для обеспечения безопасности загружаемых файлов реализована многоуровневая защита:

1.  **Валидация типов (MIME + Расширение):**
    *   `ALLOWED_IMAGE_MIME_TYPES`, `ALLOWED_VIDEO_MIME_TYPES`, `ALLOWED_FILE_EXTENSIONS`: Белые списки разрешенных типов.
    *   `DANGEROUS_FILE_EXTENSIONS`, `DANGEROUS_MIME_TYPES`: Черные списки заведомо опасных типов (например, `.exe`, `.html`, `application/javascript`).
    *   Функция `determine_upload_content_type` проверяет файл по обоим спискам. Загрузка GIF-файлов запрещена намеренно.
2.  **Ограничение размера:** `MAX_UPLOAD_SIZE = 20 * 1024 * 1024` (20 MB). Файлы большего размера отвергаются.
3.  **Сохранение:** Файл сохраняется на диск в директорию `/uploads` с именем `{secrets.token_hex(16)}{суффикс}`, что предотвращает уязвимости path traversal и перезапись существующих файлов.
4.  **Санитизация имен:** Оригинальное имя файла фильтруется функцией `sanitize_filename` для удаления потенциально опасных символов.
5.  **Отдача файлов:** Эндпоинт `GET /attachments/{message_id}` сначала проверяет валидный `room_token`, а затем строит абсолютный путь к файлу (функция `build_attachment_path`) и проверяет, что он находится внутри директории `UPLOADS_DIR`.

#### 6. Клиентское приложение

*   **Управление состоянием:** Данные аутентификации (`token`, `username`, `is_admin`) сохраняются в `localStorage`. Текущее состояние интерфейса (активная комната, язык, язык) хранится в глобальных переменных JS.
*   **Жизненный цикл:**
    1.  При загрузке вызывается `initializeSession()`. Проверяется наличие токена, и если он есть — загружается профиль пользователя через `GET /me`.
    2.  После успешной аутентификации клиент переходит в состояние "залогинен" и подгружает список комнат с автообновлением.
    3.  При входе в комнату (`joinRoom`) устанавливается WebSocket-соединение и загружается история сообщений.
    4.  Все входящие WebSocket-сообщения обрабатываются единой функцией `addMessage`, которая рендерит их в DOM.
*   **Интернационализация (i18n):** Реализована через объект `translations` в `script.js`. Функция `setLanguage` переключает язык и вызывает `applyTranslations` для обновления всего текста на странице и перерисовки комнат. Язык сохраняется в `localStorage`.

#### 7. Тестирование

*   **Фреймворк:** `pytest`.
*   **Фикстуры (`conftest.py`):**
    *   `session_factory`: Создает **in-memory** базу данных SQLite и настраивает ее для тестов.
    *   `client`: Создает экземпляр `TestClient` для FastAPI приложения с переопределенными зависимостями (БД, директория загрузки). После каждого теста производится очистка глобального состояния (комнаты, соединения, rate-limiter).
    *   `test_api.py`: Содержит интеграционные тесты для основных сценариев использования: регистрация/логин, создание комнаты, присоединение, обмен сообщениями через WebSocket, проверка прав администратора при удалении сообщений и комнат.
    *   `test_auth.py`: Модульные тесты для функций хеширования паролей и создания/верификации JWT.
